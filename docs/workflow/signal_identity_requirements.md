# Signal Identity Requirements

**Status: CURRENT WORKFLOW REQUIREMENT.** Signal identity in review-only roadway_graph outputs must use `stable_signal_id` as the canonical internal key. Source identifiers remain preserved as provenance fields, but no workflow should require HMMS `GLOBALID` to exist before a signal can receive a project-stable internal ID.

## Bounded Question

This note defines the identity contract needed for review-only represented and recovered signal records. It does not promote recovered records, assign crashes/access, calculate rates/models, or change active scaffold outputs.

## Required Fields

Every represented or recovered signal-level output should preserve:

- `stable_signal_id`
- `source_signal_id`
- `source_layer`
- `source_system`
- `GLOBALID` where available
- `OBJECTID` where available
- `asset_id` where available
- `region_signal_id` where available
- locality or region fields where available
- signal geometry hash where geometry is available
- source identity hash
- stable ID construction method
- stable ID confidence

## Construction Hierarchy

1. Preserve an existing `stable_signal_id`.
2. Otherwise construct from source layer, best available source identifiers, and coordinate or geometry hash.
3. For records lacking `GLOBALID`, use source layer, available local IDs, and geometry hash.
4. Never rely on display name alone.

Missing `GLOBALID` is a source-identifier limitation, not automatically a map-review risk. Map review should be triggered by actual duplicate, sibling, scaffold-overlap, or complex multi-signal evidence.
