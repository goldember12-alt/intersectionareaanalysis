# Map Review Workflow

Status: planning convention for future QGIS review packages. This document does not create map layers or move existing outputs.

## Bounded Question

Define a repeatable package layout for manual QGIS review of roadway-graph scaffold, leg coverage, access catchments, and later crash catchments.

## Package Root

Preferred root:

`work/output/roadway_graph/map_review/current/<review_name>/`

Use a descriptive review name, for example:

- `expanded_universe_physical_leg_review`
- `expanded_universe_access_catchment_review`
- `two_leg_under_capture_review`
- `five_plus_over_split_review`

When a review cycle is superseded or complete, move the whole package to:

`work/output/roadway_graph/map_review/archive/<review_name>_<YYYYMMDD>/`

Do not archive or move packages without an explicit cleanup task.

## Recommended Package Contents

Each package should include:

- `README.md`: review purpose, bounded questions, layer list, symbology notes, and reviewer instructions.
- `layers/`: spatial layers.
- `styles/`: QML style files where practical.
- `queues/`: CSV review queues and nonspatial summary tables.
- `project/`: optional QGIS project file if feasible.
- `manifest.json`: producer script, source inputs, created timestamp, CRS, and file inventory.

Each QGIS review package should include spatial context by default:

- basemap instructions in the README, or a QGIS project with a basemap/XYZ placeholder;
- a full preserved source roadway layer, preferably the normalized Travelway source used before graph-edge construction;
- a derived graph edge/reference roadway layer where useful, clearly labeled as derived and not a replacement for source Travelway;
- review layers specific to the bounded question;
- a checklist CSV for notes and decisions;
- a layer inventory.
- a CRS report in the README, manifest, or QA output that states source CRS, output CRS, and whether reprojection occurred.

Do not download basemap tiles into the repo. Use an XYZ connection such as OpenStreetMap from QGIS when needed.

Preferred spatial format:

- GeoPackage (`.gpkg`) for primary review layers when possible.
- GeoJSON for lightweight examples, small queues, or web-friendly spot checks.
- CSV only for nonspatial review queues or point tables with explicit coordinate fields.

## Expected Layer Families

Future packages should be assembled from review outputs, not from active-output mutation.

Expected layers include:

- Signal points.
- Candidate physical legs.
- Candidate bin geometries.
- Candidate branch/carriageway sub-branches.
- Two-leg under-capture examples.
- Five-plus over-split examples.
- Access catchment examples.
- Untyped access source/capture examples.
- Typed v2 access source/capture examples.
- Later crash catchment examples after crash-catchment design exists.

## Review Questions

Each package README should ask concrete questions, such as:

- Are apparent two-leg signals actually missing cross-street legs?
- Are five-plus candidate branches true complex intersections or divided/parallel over-splits?
- Do physical-leg clusters match reviewer expectations at typical four-leg and T-intersections?
- Are candidate bins extending along the defensible roadway legs?
- Are access catchments spatially plausible for untyped broad access counts?
- Are typed v2 access points too sparse for spatial-only assignment?
- For later crash work, do proposed crash catchments respect the reviewed scaffold?

## Styling Convention

Where possible:

- Use stable layer names matching the review queue names.
- Use distinct styles for signals, legs, bins, access points, and catchments.
- Style under-capture flags and over-split flags differently.
- Keep partial-window coverage visible rather than hiding it.
- Include labels only where useful for review, such as `signal_id`, queue class, or physical-leg count.

## Producer Expectations

A map-review producer script should:

1. Read from `work/output/roadway_graph/review/current/...` or stable graph tables.
2. Avoid reading crash records unless the package is explicitly a crash-catchment review.
3. Write only to `work/output/roadway_graph/map_review/current/<review_name>/`.
4. Include a manifest and README.
5. Keep layer row counts and signal counts separate.
6. Preserve uncertainty fields and review flags.
7. Avoid promoting review decisions into active outputs.

## Non-Goals

Map-review packages are not active analytical outputs by themselves.

Do not use them to:

- Promote candidates.
- Modify active scaffold outputs.
- Assign access or crashes.
- Calculate rates.
- Run models.
- Make policy claims.

Review results should be incorporated only by a later bounded implementation pass with explicit QA.
