Status: active review requirement for future scaffold, access, and crash/catchment outputs.

## Bounded Question

Preserve enough source lineage for every roadway scaffold/bin/access/catchment row to trace back to stable source Travelway and signal records without relying on package-local GeoPackage `fid` values.

## Required Fields

Future scaffold, access, and crash/catchment outputs should carry:

- `stable_travelway_id`
- `stable_signal_id`
- `source_signal_id`
- `stable_bin_id`
- `source_layer`
- `source_route_id`
- `source_route_name`
- `source_route_common`
- `source_measure_start`
- `source_measure_end`
- `source_feature_local_fid`
- `geometry_hash`
- `lineage_match_method`
- `lineage_confidence`

## Stable Travelway ID Rule

Build `stable_travelway_id` from:

`source_layer | source_route_id | source_route_name | source_route_common | source_measure_start | source_measure_end | geometry_hash`

The geometry hash is a reproducibility check and disambiguator. It should not be the only business key.

## Workflow Implication

Lineage persistence should be fixed before further access/crash work depends on source-row-specific claims, especially when map review identifies exact source Travelway rows.

The backfill can support review-only access-to-Travelway refinement, but production crash/catchment outputs should wait for active scaffold rows to carry direct stable Travelway lineage or should clearly flag lineage as backfilled/review-only.

Generation-time persistence requirement:

- Do not rely on retrospective route-label matching for source-row claims.
- Carry `stable_travelway_id` and the source route/measure/geometry hash fields when each bin is created.
- Retain GeoPackage/local `fid` only as `source_feature_local_fid`.
- Keep `lineage_match_method` and `lineage_confidence` in downstream scaffold, access, and crash/catchment outputs.
- Report ambiguous or fanout source matches instead of forcing a stable ID.
